# Architecture

## 1. 产品定位

这是一个 Windows 桌面版“小说 AI 工作台”，不是后台管理系统，也不是通用 CRUD 工具。

核心目标是让以下四层形成连续创作工作流：

1. 设定库：长期记忆层
2. 章节梗概：阶段规划层
3. 正文：原始写作层
4. AI 建议：协作层

当前技术栈：
- Electron
- React
- TypeScript
- SQLite

当前存储拓扑：
- 单应用 SQLite
- 多 `NovelProject`

## 2. 总体分层

```text
Renderer (React UI)
  -> window.appApi.*
Preload (contextBridge + typed API)
  -> ipcRenderer.invoke(...)
Main Process (IPC handlers + services)
  -> Local Storage Layer (SQLite / app.db)
  -> AI Layer (AIService + Provider + ContextAssembler + PromptBuilder)
```

### Main Process

负责：
- Electron 窗口与菜单
- IPC 注册、参数校验、错误边界
- 本地数据查询与落库
- “只能引用前文”“只能填前文已有坑”的业务规则校验
- AI 上下文组装与模型调用编排

### Preload

负责：
- 通过 `contextBridge` 暴露白名单 API
- 保持 renderer 不直接触碰 Node 能力
- 保持 renderer 只通过 typed IPC 访问主进程

### Renderer

负责：
- 章节工作台、设定库、总览视图交互
- 自动保存与轻量反馈
- AI 候选结果确认与应用
- 只读 AI 参考层展示

### Local Storage

当前使用单个 `app.db`：
- 多项目通过 `project_id` 隔离
- `Chapter`、`Character`、`LoreEntry`、`AiSuggestion`、`StoryPit` 共用同库
- 后续可演进为“每项目一个目录 + project.db”，当前阶段不实现

### AI Layer

当前 main 侧统一通过：
- `AIService`
- `Provider` 抽象
- `ContextAssembler`
- `PromptBuilder`

约束：
- renderer 不直接拼 prompt
- renderer 不直连数据库
- 不新增“章节记忆卡”体系
- `Chapter.outline_user` 仍然是章节正式梗概，也是后续章节引用时优先读取的章节摘要

## 3. 章节工作台结构

当前章节工作台按创作流组织，顺序固定为：

1. 已关联角色 / 已关联设定
2. 历史章节引用
3. 填坑 / 埋坑层
4. 章节规划层
5. AI 参考层
6. 正文层

额外说明：
- 原先独立的“章节基本信息”卡片已取消
- 章节标题并入章节规划层
- 保存状态显示在“章节工作台”标题行右侧

### 3.1 已关联角色 / 已关联设定

- 展示当前章已关联的角色与设定
- 支持添加与移除
- 这是章节局部上下文，不替代设定库本体

### 3.2 历史章节引用

- 当前章节引用的历史章节
- 支持自动参考和手动选择
- 参考章节只能来自当前章节之前的历史章节
- 不允许引用当前章自己
- 不允许引用后续章节
- 引用时优先读取被引用章节的 `outline_user`

### 3.3 填坑 / 埋坑层

这是小说创作中的“线索管理层”，不是 issue tracker。

#### 默认交互

- 填坑与埋坑都使用“缩略内容卡片”展示
- 列表层默认只显示：
  - 缩略内容
  - 来源摘要
  - 当前状态
  - “...” 详情按钮
- 点击“...”后，再进入详情查看 / 编辑
- 列表层不再常驻大文本编辑区
- 列表层不再堆叠保存 / 取消按钮

#### 埋坑

- 记录本章新增、后文需要回应的坑 / 伏笔 / 未解问题 / 线索
- 不再使用“本章是否埋坑”总开关
- 一个章节是否埋坑，由该章是否实际新增了至少一条坑自然体现
- “新增坑”入口进入新增坑面板
- 新增坑面板至少支持两种方式：
  - AI 推荐候选
  - 直接手动创建
- AI 只是新增时的辅助来源，不是坑的长期用户可见标签

#### 填坑

- 当前章可选择并回应前文已有坑
- 只能选择当前章节之前已经存在的坑
- 不能选择未来章节产生的坑
- 章节坑和作者手动设定坑都可被填
- 当前章节中对“已填坑”内容的编辑，直接修改该坑的全局内容

### 3.4 章节规划层

章节规划层不是纯展示层，而是当前章的写作目标层，位于“填坑 / 埋坑层”下方、“AI 参考层”上方。

当前包含：
- 章节标题
- 本章目标
- 本章梗概（`outline_user`）
- 章末钩子 / 下一章引子

AI 辅助：
- 章节标题：AI 生成候选
- 本章目标：AI 生成候选
- 本章梗概：AI 提炼候选

规则：
- AI 返回候选，不直接覆盖字段
- 用户确认后才写回

### 3.5 AI 参考层

- 只读
- 这是给作者看的参考上下文视图
- 不等于真正发给模型的 payload

当前动态汇总：
- 本章目标
- 本章梗概
- 章末钩子 / 下一章引子
- 已关联角色
- 已关联设定
- 参考章节
- 本章填坑
- 本章埋坑

### 3.6 正文层

- 原始写作正文

## 4. 层级语义

章节工作台的语义应为：

1. 先整理本章关联角色 / 设定
2. 再整理历史章节引用
3. 再整理本章要填哪些坑、埋哪些坑
4. 然后把这些线索转化为当前章的写作目标，即章节规划层
5. AI 参考层对这些内容做只读汇总，供 AI 进入正文前参考
6. 最后进入正文创作

这意味着：
- 填坑 / 埋坑层位于章节规划层上方
- 章节规划层位于 AI 参考层上方

## 5. 统一上下文组装层

### 5.1 设计目标

统一 AI 能力的输入边界，避免不同功能各自临时拼 prompt。

`AI 参考层` 负责给作者看；
`ContextAssembler + PromptBuilder` 负责给模型用。

### 5.2 ContextAssembler 输入

至少包括：
- 项目基础信息
  - `project.id`
  - `project.title`
  - `project.description`
- 当前章节基础信息
  - `chapter.id`
  - `chapter.index_no`
  - `chapter.title`
  - `chapter.goal`
  - `chapter.outline_user`
  - `chapter.next_hook`
  - `chapter.content`
- 已关联角色
- 已关联设定
- 参考章节
  - 严格只能来自前文章节
- 本章填坑
- 本章埋坑
- 当前任务类型

### 5.3 ContextAssembler 输出

至少包括两类：

1. 给 UI 展示的参考文本
- 用于 AI 参考层或调试查看

2. 给模型调用的结构化上下文对象
- 例如 `ChapterAiContext`
- 用于后续 `PromptBuilder` 和 provider 调用

### 5.4 PromptBuilder 职责

- 接收结构化 `ChapterAiContext`
- 按任务类型生成模型输入
- 不负责查库
- 不负责 UI 状态

## 6. 参考章节系统（Reference Chapters）

### 6.1 设计目标

1. 当前章支持自动参考最近若干前文章节
2. 当前章支持手动选择任意历史章节作为参考
3. 自动参考与手动参考可以并存
4. 手动 / 固定引用不应被自动推荐直接覆盖
5. AI 参考层与 AI 上下文都会纳入参考章节
6. 参考内容优先读取被引用章节的 `outline_user`

### 6.2 业务规则

- 参考章节只能来自当前章节之前的历史章节
- 自动推荐和手动选择都必须遵守“只能引用前文，不能引用后文”
- 不允许引用当前章节自己
- 不允许引用未来章节
- 若被引用章节 `outline_user` 为空，实现层可以降级展示，但不新增新的摘要实体

## 7. 填坑 / 埋坑系统

### 7.1 设计目标

1. 显式管理本章新增的坑
2. 显式管理本章回应的旧坑
3. 为长篇创作提供连续性管理，而不是通用任务流
4. 为 AI 参考层和后续 AI 能力提供稳定上下文

### 7.2 产品语义

- 坑只保留两类核心来源语义：
  - 章节坑
  - 作者手动设定坑
- AI 不再作为坑的长期用户可见分类标签
- AI 只是新增坑时的辅助能力

### 7.3 全局唯一真源

- `StoryPit` 是全局实体
- 当前章节中编辑坑内容，修改的是 `StoryPit.content` 本身
- 因此以下位置必须联动一致：
  - 当前章节填坑列表
  - 当前章节埋坑列表
  - 全部坑内容总览
  - AI 参考层中的【本章填坑】/【本章埋坑】

## 8. 项目级总览

### 8.1 章节梗概总览（Chapter Outline Overview）

这是项目级查阅能力，不是新实体。

目标：
- 集中查看项目内所有章节的 `outline_user`
- 按章节顺序浏览
- 快速看到：章节序号、章节标题、本章梗概、更新时间
- 某章 `outline_user` 更新后，总览在当前会话中即时反映

约束：
- 数据直接来自 `Chapter.outline_user`
- 不新增独立摘要存储
- 不复制新的章节摘要数据

### 8.2 全部坑内容总览

这是项目级“线索总览”，直接读取 `StoryPit`。

结构：
- 分成两个大区块：
  - 章节坑
  - 作者手动设定坑
- 章节坑按章节分组展示，例如：
  - 第 1 章《风》
  - 第 2 章《云》

默认展示：
- 缩略内容
- 当前状态
- 若已填，显示填于第几章
- “...” 详情按钮

展开详情后：
- 查看完整内容
- 编辑完整内容
- 删除

目标：
- 让作者一眼看出每章有哪些坑
- 看出哪些坑已填、哪些未填
- 看出自己手动设定了哪些坑

## 9. 关键流程

### 9.1 写作流程

1. 选择项目与章节
2. 整理本章关联角色 / 设定
3. 添加或自动推荐历史章节引用
4. 选择本章填坑，并按需要新增本章埋坑
5. 基于这些素材形成章节规划：标题 / 目标 / 梗概 / 钩子
6. AI 参考层自动汇总上述内容
7. 进入正文创作或触发 AI 能力

### 9.2 AI 参考构造流程

1. 读取当前章节规划字段
2. 读取当前章节关联的角色与设定
3. 读取当前章节的参考章节列表
4. 读取当前章节填坑 / 埋坑信息
5. `ContextAssembler` 生成结构化 `ChapterAiContext`
6. `PromptBuilder` 生成模型输入
7. UI 侧 AI 参考层同步展示用户可读参考文本

### 9.3 标题 / 目标 AI 生成流程

1. 用户点击“标题 AI 生成”或“目标 AI 生成”
2. main 查询项目、章节、关联角色 / 设定、参考章节、填坑 / 埋坑信息
3. `ContextAssembler` 组装统一上下文
4. `PromptBuilder` 按任务类型生成模型输入
5. provider 返回候选文本
6. renderer 展示确认 UI
7. 用户确认后才写回对应字段

### 9.4 AI 提炼梗概流程

1. 用户点击“AI 提炼梗概”
2. 系统先校验当前正文是否为空
3. main 查询项目、章节、关联角色 / 设定、参考章节、填坑 / 埋坑
4. `ContextAssembler` 组装统一上下文
5. `PromptBuilder` 生成模型输入
6. `AIService.summarizeChapterFromContent` 调用 provider
7. 返回候选梗概
8. renderer 展示确认 UI
9. 用户确认后才写回 `outline_user`
10. 应用成功后联动刷新：
  - 当前章节详情
  - AI 参考层
  - 章节梗概总览
  - 引用本章 `outline_user` 的位置

### 9.5 坑位编辑联动流程

1. 用户在章节层或项目级总览中打开某条坑详情
2. 编辑该坑完整内容
3. 用户确认保存
4. main 更新 `StoryPit.content`
5. renderer 刷新：
  - 当前章节填坑 / 埋坑列表
  - 全部坑内容总览
  - AI 参考层中的坑位摘要

### 9.6 项目级查阅流程

1. 作者打开“章节梗概总览”
2. 按章节顺序快速回看所有章节的 `outline_user`
3. 作者打开“全部坑内容总览”
4. 先按“章节坑 / 作者手动设定坑”分区查阅
5. 在“章节坑”区块内按章节分组查看每章产生的坑
6. 展开某条坑后查看或编辑完整内容

## 10. 当前目录建议

```text
src/
  main/
    ai/         # AIService / providers / ContextAssembler / PromptBuilder
    db/         # SQLite access
    main.ts     # Electron main + IPC
  preload/      # contextBridge API
  renderer/     # React UI
  shared/       # IPC types / shared contracts
```

## 11. 当前阶段不做

- 章节记忆卡 / 摘要卡新体系
- issue tracker 式任务系统
- 一致性检查引擎
- 导入导出
- 云同步
- 富文本编辑器
- embedding / 复杂语义检索

## 12. 本轮补充：填坑入口与埋坑入口对齐

- 章节工作台中的“填坑”和“埋坑”都采用按钮入口，不再保留“填坑列表底部下拉框 + 按钮”的旧交互。
- “填坑”点击后打开选择面板，先选前文旧坑，再可在确认前微调坑内容。
- “埋坑”点击后打开新增面板，可采用 AI 候选或直接手动创建。
- 两侧都保持“缩略卡片列表 -> 弹窗详情 / 编辑 / 确认”的一致节奏，避免一侧是卡片、一侧是表单的失衡布局。

## 13. 本轮补充：AI 提炼摘要避免回灌套娃

- `AI 提炼梗概 / 摘要` 的产品语义调整为：在章节完成或接近完成时，根据“当前正文”生成一版更新后的章节摘要候选。
- 这条链路的核心输入是当前章节正文，不再把现有 `Chapter.outline_user` 作为提炼任务的主要输入回灌给模型。
- 这样可以避免“上一次已应用的摘要再次参与下一轮摘要生成”的套娃现象。
- `Chapter.outline_user` 仍然是当前章的正式摘要 / 梗概真源；AI 只返回候选，用户确认后才覆盖。

## 14. 本轮补充：摘要生成可回退到 AI 参考

- 当当前正文为空时，若 AI 参考层已有足够上下文（目标、钩子、关联角色 / 设定、参考章节、坑位等），仍可生成候选摘要。
- 这类候选摘要本质上是“基于当前章节规划与上下文的参考摘要”，用于写作起步阶段辅助作者。
- 但在该任务中，现有 `Chapter.outline_user` 仍然不进入模型输入，避免摘要自我回灌。
