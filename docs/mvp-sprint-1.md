# MVP Sprint 1

## 1. 状态标识规则
- `[主流程]`：当前用户已经能在首页、章节管理页、写作页主路径中直接操作到的范围。
- `[兼容保留（非当前主流程）]`：能力仍然在代码、数据层、IPC 和 AI 服务中存在，但当前主界面还没有把它重新接回首页主流程。
- `[已废弃/已删除]`：只有真正不再保留时才使用；如果还在代码里，只是不作为主流程，必须标记为兼容保留。

## 2. 当前阶段定位
当前项目仍处于 MVP 基线阶段，但这里的 MVP 不是“只剩某一轮局部任务”，而是当前整个小说 AI 工作台已经累计形成的最小可用产品面。

当前 MVP 的目标是：
- 保证 Windows 桌面端主流程可用
- 保证本地单库、多作品、章节与回收站可用
- 保留 AI、知识、坑位、Suggestion、历史章节引用等结构化能力的数据与服务边界
- 在继续重构代码结构时，不破坏已有产品语义

## 3. [主流程] 当前主流程 UI 范围

### 3.1 [主流程] 桌面壳与启动
- Electron + React + TypeScript 工程已可运行
- preload 已接通
- `app.init` 已接通真实初始化
- 本地数据库可正常初始化与恢复数据

### 3.2 [主流程] 首页与作品入口
- 左侧只保留：`作品管理`、`开书灵感`、`作品回收`
- 作品管理支持展开作品列表
- 作品回收支持展开已删除作品列表
- 开书灵感支持弹窗入口
- 首页空状态支持：`开书灵感`、`去写作`
- 右上角支持：`创建新书`
- 作品概览已收为紧凑横向信息带，而不是巨型卡片
- 首页渲染结构已开始按组件拆分：`HomeHeader`、`HomeEmptyState`、`ProjectSummaryStrip`、`DeletedProjectSummaryStrip`
- `HomePage` 当前以页面装配为主，首页大块 JSX 不再继续集中堆叠在单文件中

### 3.3 [主流程] 作品回收
- 删除作品进入作品回收
- 恢复作品可回到作品管理
- 永久删除可从作品回收清除

### 3.4 [主流程] 章节管理与章节回收站
- 可从首页进入章节管理页
- 支持章节列表、章节回收站、搜索章节、新建章节
- 删除章节进入章节回收站
- 恢复章节后重新进入当前作品章节列表
- 章节管理页渲染结构已开始按组件拆分：`ChapterHeaderBar`、`ChapterTabs`、`ChapterToolbar`、`ChapterFilterBar`、`ChapterListRow`、`ChapterRecycleRow`、`ChapterEmptyState`
- `ChapterManagementPage` 当前以页面装配为主，章节页大块 JSX 不再继续集中堆叠在单文件中

### 3.5 [主流程] 写作页
- 支持轻量写作页入口
- 支持章节标题和正文编辑
- 支持保存状态展示
- 支持自动保存配置展示

## 5. 当前 MVP DoD（累计口径）
满足以下条件，可认为当前 MVP 基线成立：
- `npm run build` 通过
- 应用启动后 `app.init` 正常
- 可创建作品、查看作品、删除作品、恢复作品、永久删除作品
- 可创建章节、删除章节、恢复章节、永久删除章节
- 可进入章节管理页与写作页
- preload / IPC / DB / repository 调用链保持通畅
- 文档描述当前整个项目真实状态，而不是单轮局部说明

## 6. 当前明确不做 / 未完成部分
- 不做云同步
- 不做富文本编辑器
- 不做复杂 migration 系统
- 不把所有已保留的 AI / 知识 / 坑位 / Suggestion / 历史引用能力一次性重接到首页
- `useWorkspaceController.tsx` 已继续按 `hooks/workspace/*` 拆到页面级 controller（首页 / 章节管理页 / 写作页）与局部状态 hook，顶层 controller 当前主要保留共享数据加载、自动保存链路与页面装配职责
- 部分弹窗和页面仍在视觉细节收敛中

## 7. 当前主要技术债
- `useWorkspaceController.tsx` 仍然集中度较高
- `mappers.ts` 与 `entity-loaders.ts` 仍偏大
- 当前主流程 UI 与已保留结构化能力之间，仍有重新整合工作
- 部分视觉细节是迭代修补后的结果，仍需统一整理
- renderer 侧虽然已完成首页第一轮组件抽取，但首页菜单与章节管理页仍有进一步抽公共组件的空间
- renderer 侧已完成首页、章节管理页与写作页 controller 的第一轮拆分，后续重点转向继续压缩顶层 controller 的共享编排责任

## 8. 下一阶段建议
1. 继续拆 renderer controller，降低顶层状态集中度
2. 继续把 AI、知识、坑位、Suggestion 重新接回简化后的主流程 UI
3. 统一章节回收站、写作页的交互细节
4. 继续清理文档，让每次变更都按“新增 / 修改 / 废弃 / 删除”作用在整体项目说明上

- 本文件明确区分：`[主流程]` 与 `[兼容保留（非当前主流程）]`。
- 今后新增能力时，应补进当前范围；修改已有规则时，应直接更新对应条目；只有真正删除的能力，才从文档中移除。
